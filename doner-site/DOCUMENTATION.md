# Исчерпывающая Техническая Документация ПРОЕКТА "Doner-Site"

Настоящий документ содержит подробнейшее описание каждого модуля, компонента, маршрута роутинга и сущности базы данных, используемых в данном проекческом приложении для сервиса заказа еды.

---

## 1. ИНФРАСТРУКТУРА И ДЕПЛОЙ (DOCKER)

Система собирается через `docker-compose.yml` и содержит четыре ключевых сервиса:

1. **База данных (`db`)**: Образ `postgres:13-alpine`. 
   - Выставляет порт сервера СУБД 5432.
   - Развернут с использованием `persist` volume для гарантии не удаляемости данных.
2. **Backend-сервер (`backend`)**: Образ на базе `node:20-alpine`. 
   - Запускается на `NestJS` и "под капотом" `Express`. 
   - Папка с фотографиями `./BD_PHOTO` пробрасывается как volume.
3. **Frontend-сервер (`frontend`)**: Образ на базе Nginx. 
   - Внутри Dockerfile происходит сборка React (`npm run build`) и копирование папки `build` в web-root директорию Nginx.
4. **Proxy / Reverse Proxy (`nginx`)**: Дополнительный сервисный слой, который разделяет трафик. 
   - Все запросы, начинающиеся на `/api` пробрасываются на `backend:3000`.
   - Запросы `/images` пробрасываются на `backend:3000/images`.
   - Остальные запросы переправляются напрямую на скомпилированный `frontend:80`.
   - Сервис прослушивает 443 порт для работы `HTTPS` с заготовленными SSL сертификатами от `letsencrypt`.

## 2. БЭКЕНД (NESTJS API SERVER)

### 2.1 Конфигурация Приложения
- Базовый путь: `http://localhost:3000/api`
- Приложение использует `TypeOrmModule` для подключения к Postgres с автоматической синхронизацией столбцов (`synchronize: true`).
- Раздача статики выполняется через модуль `@nestjs/serve-static`, отдавая папку `BD_PHOTO` по префиксу `/images`.

### 2.2 Модуль `AuthModule` и `UsersModule`
Обеспечивают систему защиты, авторизацию (JWT) и регистрационную логику.
- **Сущность БД `User`**: Поля `id`, `email` (уникальное), `password`, `name`, `role` (user/admin), `phone`, `address`, `created_at`.
- **Стратегия Guards**:
  - `JwtAuthGuard`: Проверяет валидность токена пользователя через `PassportStrategy`. При отсутствии - `401 Unauthorized`.
  - `RolesGuard`: Допускает к эндпоинту, только если `req.user.role` соответствует требуемому (использует кастомный `@Roles('admin')` декоратор).
- **Ключевая логика `UsersService`**:
  Метод `onModuleInit`, который срабатывает при поднятии сервера, автоматически проверяет наличие аккаунта `admin@admin.ru`. Если его нет - он его генерирует (хэш-свертка пароля `bcrypt`). Это защищает от ситуации "утери контроля админки".
- **Эндпоинты (`AuthController`)**:
  - `POST /api/auth/register`: Регистрация, генерация JWT.
  - `POST /api/auth/login`: Прямой вход (проверка хэша пароля), выдает токен.
  - `GET /api/auth/me`: Аутентифицированный геттер информации текущего пользователя.
  - `POST /api/auth/update`: Обновление личного профиля.

### 2.3 Модуль `DishesModule`
Является центральным диспетчером товарной номенклатуры сайта (Меню сервиса).
- **Сущность БД `Dish`**: 
  - `id` (Primary Key).
  - `title` (строка), `desc` (строка).
  - `price` (integer), `discountPrice` (integer | null).
  - `weight` (integer).
  - `category` (integer, маппинг на клиент): 2 - Шаурма, 3 - Бртуч/Бургер, 4 - Напитки, 5 - Стрит-фуд.
  - `best_sell` (integer: 1 / 0) - флаг хита продаж.
  - `isSeasonal` (boolean) - флаг нахождения визуально в сезонном меню.
- **Ключевой фикс БД (`onModuleInit` в `dishes.service.ts`)**:
  Автоматическое выравнивание PostgreSQL счётчика `dishes_id_seq` через `SELECT setval(...)`. Решает проблему с падением 500-й ошибки при конфликтах ID после ручных правок базы админом напрямую.
- **Безопасность (`dishes.service.ts` create)**:
  В сервисе реализована 100% защита от недополучения параметров (если клиент не досылает `best_sell` - ставится 0). Также реализован надежный парсер поля `image` (зачистка дублей префикса `/images/`), чтобы URL до фото формировался правильно.

### 2.4 Модуль `OrdersModule`
Управляет жизненным циклом заказов и корзины.
- **Сущность БД `Order`**:
  Связь ManyToOne -> `User`. Содержит: JSON колонку `items` со списком блюд заказа, `status` ('new', 'cooking', 'delivering', 'completed', 'cancelled'), `total`, `paymentMethod`, `is_paid` и данные адреса.
- **Контроллер (`OrdersController`)**:
  - `POST /api/orders` - Создает заказ. Если `type != delivery`, адрес перезаписывается как "Самовывоз". Вызывает внешние сервисы `MailModule` (для оповещения) и `PaymentModule`.
  - `GET /api/orders` - Использует продвинутый постраничный функционал (`?page=1&limit=10`) - возвращает объект `{ data: [], total: N }`, либо срабатывает для всего списка без пагинации (полиморфизм функции `getAll`).

### 2.5 Модуль `MailModule`
- Вспомогательный модуль-утилита, отвечающий за отправку оповещений на email ресторана. Инстанцирует `nodemailer` с использованием `SMTP` настроек из конфигурационного `.env` (MAIL_USER / MAIL_PASS).
- Генерирует HTML разметку писем со списком товаров, их количеством (`x${item.quantity}`) и итоговой стоимостью.

### 2.6 Модуль `AdminController` (Аналитика и Метрики)
Файл `app.admin.controller.ts`. Скрыт от обычных юзеров (`@Roles('admin')`).
- `GET /api/admin/stats` - Сводит всю статистику базы: итоговая выручка, количество заказов за период ('today', 'week', 'month', 'all'), расчет распределения статусов и популярности категорий для графиков (Recharts).
- `POST /api/admin/upload` - Прием картинок через `multer` и генерация рандомного HEX-имени файла.
- `GET /api/admin/export/:type` - Подсистема выгрузки данных. Генерирует "на лету" CSV строки, приправленные BOM-маркером (`\uFEFF`) для идеальной совместимости с русским шрифтом в MS Excel. В зависимости от `:type` выдает метрики: 'users' (списки клиентов), 'orders' (смета заказов), 'revenue', 'dishes', 'status'.

## 3. ФРОНТЕНД (REACT КЛИЕНТ)

Папка `client/src`. Основано на функциональных компонентах `React Hooks`. 

### 3.1 Ядро и Система Состояния (Redux)
Менеджмент состояния полностью построен на `Redux Toolkit` (файл `store.js`).
- `dishesSlice.js`: Файл логики асинхронного скачивания блюд `fetchDish`. Здесь зашита важнейшая бизнес-логика: 
  * Если `categoryId == 6` (Сезонное меню), то Redux **не отправит** параметр категории на сервер, а запросит все блюда, чтобы не получить пустой список, так как у сезонных блюд настоящие категории 2 или 4.
  * Если `categoryId == 0` (Хиты), вычленяются только те, где `best_sell == 1`.
- `filterSlice.js`: Синхронизирует `categoryId` и `searchValue` между компонентом поиска и Главной сеткой.
- `cartSlice.js`: Хранилище корзины. Логика `totalPrice` перемножает `item.price * item.quantity`. Мутации защищены в `localStorage` (корзина не исчезнет при F5).
- `userSlice.js`: Аналог Axios interceptor-а, хранящего `token` в `window.localStorage`.

### 3.2 Архитектура Клиентских Экранов
- **`App.js` & Router**: Обёртка `BrowserRouter`. Пути: `/` (Home), `/cabinet` (Profile), `/checkout` (Корзина), `/admin/*` (Защищенная административная зона).
- **`Home.jsx` (Главная витрина)**:
  - Рендерится через компонент-фильтр. Механизм `React.useMemo` моментально фильтрует `displayedItems` без запроса к API, опираясь на стейт.
  - Каждая подкатегория имеет свой мощный приоритетный сорт: шаурма сортируется так, чтобы сначала шел 'кебаб', затем большая, затем мини. Сырные палочки сортируются математически по размеру/весу.
  - Для блюд в загрузке рисуются Skeleton компоненты.
  - Содержит `FloatingCartButton` - кнопку в левом нижнем углу для перехода в корзину.

### 3.3 Архитектура Админ-панели (`AdminLayout`)
Все админские страницы обернуты компонентом `AdminBlock` (визуальный flex-контейнер) и `AdminSidebar` (переключатель ссылок). Меню автоматически адаптируется (прячется в горизонтальную полоску) под мобильные устройства (MediaQuery <= 768px).

1. **`AdminDashboard` (Аналитика)**:
   - Делает вызов `/api/admin/stats`. Рендерит два графика (PieChart и Bar/LineChart в зависимости от выбора) библиотекой `Recharts`.
   - Содержит подблок **Экспорт данных (CSV)**. Используя функционал `Blob` и `URL.createObjectURL`, он берет ответ Axios вместе с JWT-токеном безопасности и симулирует клик по `href download` для скачивания файла отчета.
2. **`AdminDishes` (Управление Меню)**:
   - Обладает всплывающим окном `Modal`.
   - Внутри зашито строгое преобразование (sanitization) полей: `Category` реализовано в виде селекта (select options), а числа жестко скастуются парсером `Number()`. Пустая скидка отправляется как `null`, что позволяет базе Postgres не обваливаться.
3. **`AdminOrders` (Менеджер Заказов)**:
   - Взаимодействует с API через пагинацию. Сверху расположен фильтр статусов ("Новые", "Готовятся" и т.п.).
   - Снизу реализованы кнопки `< Вперед/Назад >` для смены страниц (`setState(page+-1)`).
   - Левая граница заказа (`borderLeft`) динамически подкрашивается в зависимости от статуса заказа (`getStatusColor`): от красного до зеленого.

## 4. СИСТЕМА ДИЗАЙНА И АНИМАЦИИ (SCSS + CSS)

Все стили используют BEM-подобную структуру и препроцессор `SASS/SCSS` (например, `AdminDishes.module.scss`). Изолированность стилей достигается за счет React CSS Modules (классы компилируются с хэшами).

В главном `index.css` интегрирован глобальный класс плавных эффектов, задающий премиальный вид платформе:
```css
@keyframes smoothFadeIn {
  from { opacity: 0; transform: translateY(12px); }
  to { opacity: 1; transform: translateY(0); }
}
```
- Этот `smoothFadeIn` привязан к основному контейнеру `#root > div` (смягчен кубической кривой Безье `cubic-bezier(0.16, 1, 0.3, 1)` для органичного физического старта анимации).
- За счет аппаратных триггеров `React Keys` (`key={`${categoryId}-${status}`}` в `Home.jsx`), анимация элегантно срабатывает **каждый раз**, когда вы меняете категорию.
- Для всех кнопок и инпутов прописано универсальное правило плавного затенения при `hover` и микро-анимация `scale(0.96)` при `:active`, создающие "клик-фидбек", аналогичный нативным мобильным платформам (iOS/Android).
